# PRD: 상조 근무 플랫폼 (Mutual Work Platform)

> **버전**: v0.1.0 | **작성일**: 2026-03-08 | **상태**: Draft

---

## 1. 개요 (Overview)

### 1.1 제품 비전
기업 내 팀 간 **상호 지원 근무(상조 근무)** 를 체계적으로 조율하고 관리하는 B2B SaaS 플랫폼.  
인력 부족, 업무 과부하, 부서 간 협업 비효율 문제를 해결하여 조직 전체의 리소스 활용률을 극대화한다.

### 1.2 문제 정의 (Problem Statement)

| # | 문제 | 현재 해결 방식 | 한계 |
|---|------|--------------|------|
| 1 | 팀별 업무 부하 불균형 | 이메일/메신저 수동 요청 | 추적 불가, 비효율 |
| 2 | 상조 근무 이력 관리 부재 | 엑셀 수기 관리 | 오류 多, 실시간 파악 불가 |
| 3 | 리소스 가시성 부족 | 주간 회의로 확인 | 즉각 대응 불가 |
| 4 | 정산/보상 기준 불명확 | 담당자 재량 | 공정성 이슈 |

### 1.3 목표 (Goals)

- **Primary**: 팀 간 상조 근무 요청~완료 전 과정을 단일 플랫폼에서 처리
- **Secondary**: 상조 근무 데이터 기반 리소스 최적화 인사이트 제공
- **Non-goal**: 급여/회계 시스템 직접 연동 (v1 범위 외)

## 2. 타겟 사용자 (Target Users)

### 2.1 Persona

| Persona | 역할 | 주요 Pain Point | 핵심 니즈 |
|---------|------|----------------|----------|
| **관리자 (Admin)** | HR / 운영팀장 | 부서 간 조율 비용 高 | 전사 리소스 현황 대시보드 |
| **팀장 (Manager)** | 부서 리더 | 인력 요청/제공 추적 어려움 | 요청 관리 & 이력 확인 |
| **직원 (Employee)** | 실무자 | 상조 근무 일정 혼선 | 간편한 일정 확인 & 수락 |

### 2.2 고객사 프로필
- 규모: 50~1,000명 중견 기업
- 업종: 제조, 물류, IT 서비스, 유통
- 조건: 다수의 팀/부서 운영, 시즌별 업무 편차 존재

## 3. 핵심 기능 (Core Features)

### 3.1 기능 목록

| 우선순위 | 기능명 | 설명 | 대상 |
|---------|--------|------|------|
| 🔴 P0 | **상조 요청 생성** | 팀장이 인력 지원 요청 등록 | 팀장 |
| 🔴 P0 | **요청 승인/거절** | 수신 팀장의 수락 또는 반려 처리 | 팀장 |
| 🔴 P0 | **직원 배정** | 승인된 요청에 직원 매핑 | 팀장 |
| 🔴 P0 | **상조 근무 캘린더** | 개인/팀 단위 일정 시각화 | 전체 |
| 🟡 P1 | **이력 관리** | 요청/수행 이력 조회 및 필터 | 관리자, 팀장 |
| 🟡 P1 | **알림 (Notification)** | 요청/승인/변경 이벤트 푸시 | 전체 |
| 🟡 P1 | **대시보드** | 팀별 상조 현황, 부하 지표 | 관리자 |
| 🟢 P2 | **정산 리포트** | 월별 상조 근무 시간 집계 | 관리자 |
| 🟢 P2 | **조직도 연동** | 부서/직급 구조 자동 반영 | 관리자 |
| 🟢 P2 | **API 연동** | 사내 HR 시스템 연동 지원 | 개발팀 |

### 3.2 핵심 User Flow

```
[팀장 A - 요청팀]
요청 생성 → 대상 팀 선택 → 필요 인원/기간/업무 입력 → 제출

[팀장 B - 지원팀]
알림 수신 → 요청 내용 확인 → 승인/거절 → 직원 배정

[직원]
배정 알림 수신 → 일정 확인 → 상조 근무 수행

[관리자]
전사 현황 모니터링 → 월간 리포트 확인 → 정산 처리
```

## 4. 기술 요구사항 (Technical Requirements)

### 4.1 시스템 구성 (System Architecture)

```
Frontend  : Vue 3 + TypeScript (SPA)
Backend   : Spring Boot 3.x (REST API)
Database  : PostgreSQL (운영), Redis (캐시/세션)
Auth      : JWT + OAuth2.0 (SSO 지원)
Infra     : AWS (ECS + RDS + ElastiCache)
CI/CD     : GitHub Actions
```

### 4.2 비기능 요구사항 (Non-Functional Requirements)

| 항목 | 요구 수준 |
|------|----------|
| 가용성 (Availability) | 99.9% Uptime (월 다운타임 ≤ 43분) |
| 응답 시간 (Response Time) | API P95 ≤ 300ms |
| 보안 (Security) | HTTPS 필수, RBAC 권한 체계 |
| 확장성 (Scalability) | 동시 접속 1,000명 이상 지원 |
| 다국어 | 한국어 우선, 영어 추후 지원 |

### 4.3 Integration

- **SSO**: SAML 2.0 / OAuth 2.0 (Google, Microsoft 365)
- **알림**: Email, Slack Webhook, 사내 메신저 (Webhook 방식)
- **HR 연동**: REST API (v2 이후)

## 5. 데이터 모델 (Core Data Model)

```
Company (고객사)
  └── Department (부서/팀)
        └── User (직원)
              └── WorkRequest (상조 요청)
                    ├── RequestStatus: PENDING | APPROVED | REJECTED | COMPLETED
                    ├── WorkAssignment (직원 배정)
                    └── WorkLog (근무 이력)
```
